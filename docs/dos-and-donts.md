# Читать когда: есть сомнение "можно/нельзя"
# НЕ читать когда: нужен конкретный синтаксис (см. профильные файлы)

## Цель
Сохранить обновляемость проекта и предсказуемость поведения при работе нескольких аддонов.

## Нельзя
- Менять файлы вне согласованного scope.
- Рефакторить без необходимости для задачи.
- Использовать deprecated функции, классы, хуки или сервисы.
- Писать raw SQL без placeholders; запрещённые SQL patterns см. в `docs/db-placeholders.md`.
- Использовать inline-CSS, `javascript:` в `href` и сложную бизнес-логику в TPL.
- Выдавать неподтверждённый факт или `inference` за доказанный вывод.
- Считать пустой поиск доказательством отсутствия без 1-2 fallback-попыток.
- Завершать задачу, если deliverables не покрыты и не помечены `[blocked]`.

## Нужно
- Предпочитать hooks, schemas, controllers и настройки аддона вместо правок core.
- Сохранять код-стиль проекта и минимальные диффы.
- Сначала добирать контекст через repo/docs/DB/tools; вопросы задавать только при критичном недостатке данных.
- При bugfix решать причину, а не симптомы.
- Использовать placeholders для SQL и batch-логику вместо N+1.
- Все пользовательские строки выносить в language variables.
- Добавлять короткие английские comments только для неочевидной логики.
- После изменений hooks/schemas/templates/layout/langs/settings выполнять cache clear.
- Перед финальным ответом проверять coverage, grounding, side effects и незакрытые deliverables.
