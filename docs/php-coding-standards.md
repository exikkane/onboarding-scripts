## Назначение
Единый стиль PHP-кода для предсказуемого поведения и поддержки.

Все новые PHP-файлы создавать с обязательной шапкой в начале файла:
`/***************************************************************************`
`*                                                                         *`
`*                             Larionov.tech                               *`
`*                          https://larionov.tech                          *`
`*                                                                         *`
`***************************************************************************/`
- type declarations в новых функциях. как return type declaration, так и arguments.

## Именование
- Переменные: `snake_case`.
- Булевы: `is_`, `has_`, `use_`.
- Списки: суффикс `_list`.
- Функции аддона: `fn_[addon_id]_[action]`.

### Шаблон именования функций по назначению
- Получение данных: `fn_[addon]_get_[entity]()`.
- Обновление/сохранение: `fn_[addon]_update_[entity]()`.
- Удаление: `fn_[addon]_delete_[entity]()`.
- Утилиты: `fn_[addon]_[verb]_[target]()`.
- Hook handlers: `fn_[addon]_[hook_name]` (имя должно совпадать с именем hook).

## Структура
1. Не делать глубокую вложенность без необходимости.
2. Повторяющуюся логику выносить в отдельные функции.
3. Явно приводить типы для входящих данных.
4. Не меняй код-стиль проекта
5. Короткие inline comments на английском только для неочевидной логики
6. Избегать N+1 запросов
7. Код должен быть документирован и соответствовать стандартам PSR-12
8. Не рефакторь без явной необходимости
9. Для `enum`/`class`/`interface`/`trait` использовать `use`-импорты; не оставлять fully qualified names в теле кода без крайней необходимости

## Core API compatibility
- Не использовать функции, классы, хуки и сервисы, помеченные в PHPDoc или документации как `@deprecated`.
- Если существующий код использует deprecated API и задача затрагивает этот участок, заменить вызов на актуальный core API минимальным диффом.
- Перед добавлением нового вызова core API проверять объявление функции/класса и ближайшие примеры использования в core.

## Сравнения и проверки
- Использовать строгие сравнения `===`.
- Использовать явные проверки вместо неочевидных условий.
- Не полагаться на неявные преобразования типов.

## Ошибки и диагностика
- Критичные ситуации оформлять исключениями соответствующего уровня.
- Для локальной отладки использовать `fn_print_r` и `fn_print_die`, затем удалять.
